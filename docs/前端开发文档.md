## 0. 一分钟速览

MusicMaster = 一体化音乐处理,四个互相独立的能力,一套 FastAPI Web 应用串起来:

| Tab | 能力 | 计算 | 后端入口模块 | 调用方式 |
|---|---|---|---|---|
| 🎼 互译 | 简谱 ⇄ 五线谱 双向无损 + 导入 | **CPU**,本进程 | `musicmaster.convert.convert` | 直接函数调用 |
| 🎤 记谱 | 音频 → 五线谱 + 简谱 + 逐音可信度 | **CPU**,本进程 | `musicmaster.transcribe.autopilot` | 直接函数调用 |
| 🔊 拆声 | 混音 → 人声/伴奏/去和声/降噪 | **GPU**,子进程 | `musicmaster.separate.pipeline` | **子进程** → 分离 venv(`.venv-sep`) |
| 🎚️ 重塑 | 跑调清唱 → 在调+干净+仍是你 | **GPU**,子进程 | `musicmaster.vocal.pipeline` | 本进程调 `two_stage`,**其内部再子进程**到 GPU venv |

**整体数据流(四个部门完全一致)**:

```
浏览器(设计稿 + 部门 JS)
  │  ① POST /api/<fn>  (multipart 表单 + 文件)
  ▼
FastAPI(web/server.py)
  │  落盘上传 → JobManager.submit(runner, ...) → 立即返回 { job_id }
  ▼
线程池 worker(web/jobs.py)跑 runner(web/runners.py)
  │  runner 复用核心模块(convert/transcribe/separate/vocal),更新 stage/progress
  ▼
浏览器  ② GET /api/job/<id>  每 800ms 轮询,直到 status=done|error
        ③ GET /api/file/<id>/<name>  下载/预览产物
```

**关键事实**:服务跑在 CPU 主环境(`.venv`)。互译/记谱与它同环境 → runner 内**直接 import 调用**;
拆声是 GPU 重依赖、独立 venv → runner **子进程**调 `.venv-sep`;重塑由本进程调 `vocal.pipeline.two_stage`,
**其内部再起 GPU 子进程**到修音/换音色 venv。所以**互译 / 记谱开箱即用**,拆声 / 重塑需先装 GPU venv(见 §6)。

**前端最小心智模型**:前端 = 收集输入 → `MM.run('<fn>', formData, onProgress)` → 渲染 `job.result`。
不必懂任何算法;所有部门模块都建立在 `static/js/api.js`(`window.MM`)之上,因此交互一致、各模块很薄。

## 1. 总览与启动

### 1.1 启动方式(四选一)

| 方式 | 命令 | 适用 |
|---|---|---|
| **双击启动器** | `启动.bat`(Windows) / `bash start.sh`(macOS/Linux) | 用户首选;自动 `chcp 65001` + 设 `MUSICMASTER_OPEN_BROWSER=1` 弹浏览器 |
| **顶层 CLI** | `musicmaster gui`(装包后) | 等价于起 `web.server`,会设 UTF-8 + 自动开浏览器 |
| **直接跑模块** | `.venv\Scripts\python.exe -m musicmaster.web.server` | 开发/预览首选;**默认不弹浏览器**(没设 `MUSICMASTER_OPEN_BROWSER`) |

启动后服务监听 **`http://127.0.0.1:7860`**(被占用自动顺延 7861…7910)。关闭运行窗口 = 停服务。

> `启动.bat` / `start.sh` 仅做三件事:进到仓库根、检查 `.venv` 是否存在、`set MUSICMASTER_OPEN_BROWSER=1`
> 后跑 `python -m musicmaster.web.server`。`musicmaster gui`(`musicmaster/cli.py`)也只是带这两个环境变量
> `subprocess.call` 同一条命令。**它们都没有自己的逻辑,真正的服务全在 `web/server.py`。**

### 1.2 启动相关环境变量

| 环境变量 | 作用 | 默认 |
|---|---|---|
| `MUSICMASTER_HOST` | 监听地址 | `127.0.0.1` |
| `MUSICMASTER_PORT` | 首选端口(被占用自动顺延;非数字回退 7860) | `7860` |
| `MUSICMASTER_WORKERS` | 线程池并发数(见 §3.4) | `2` |
| `MUSICMASTER_OPEN_BROWSER` | 设为 `1` 才在启动 1.5s 后自动开浏览器 | 未设(不弹) |

### 1.3 路由一览(`web/server.py`)

| 方法 + 路径 | 作用 |
|---|---|
| `GET /` | 设计稿首页(`StaticFiles(html=True)`) |
| `GET /js/*` `GET /fonts/*` … | 静态资源(JS / 自托管字体) |
| `POST /api/convert` `transcribe` `separate` `vocal` | 提交任务(multipart),返回 `{ job_id }` |
| `GET /api/job/{id}` | 轮询任务状态/结果(JSON) |
| `GET /api/file/{id}/{name}` | 下载/预览某任务产物(防目录穿越) |
| `GET /api/health` | 健康检查 `{ ok, service, static }` |
| `GET /api/docs` `GET /api/openapi.json` | FastAPI 自动文档(联调时可直接点开试 API) |

> ⚠️ **挂载顺序**:静态站用 `app.mount("/", StaticFiles(...))` 挂在根,**必须放在所有 `/api` 路由之后**,
> 否则会把 `/api/*` 一并吞掉。`server.py` 已是正确顺序;新增路由也要加在 `mount` 之前。
> 若 `static/` 目录缺失,根路由回退成一条说明 JSON(提示把 design 复制到 `web/static/`)。

## 2. 后端 API 契约

> 这是换前端 / 加功能的「合同」。前端只要照此提交与解析,后端逻辑不必动。

### 2.1 四个提交端点(`POST /api/<fn>`,multipart/form-data)

所有提交端点都**立即返回** `{ "job_id": "<hex>" }`(不阻塞;真正干活在后台线程)。表单字段:

| 端点 | 字段 | 类型 | 默认 | 含义 |
|---|---|---|---|---|
| `/api/convert` | `file` | 文件 | (必填) | 谱文件:`.jianpu`/`.musicxml`/`.xml`/`.mxl`/`.mid`/`.midi`/`.abc` |
| | `direction` | 文本 | `j2s` | `j2s`=简谱→五线谱 · `s2j`=五线谱→简谱 |
| | `key` | 文本 | `""` | 主音(仅 `s2j` 用,如 `G`);留空自动 |
| `/api/transcribe` | `audio` | 文件 | (必填) | 清唱/单旋律音频 |
| | `engine` | 文本 | `crepe` | `crepe` / `basic-pitch` / `bytedance`(见 §5.1) |
| | `key` | 文本 | `""` | 调(1=主音),如 `C`;留空自动定调 |
| `/api/separate` | `audio` | 文件 | (必填) | 整首混音 |
| | `stages` | 文本 | `1,2,3` | 处理段:`1`=人声/伴奏 · `2`=去和声 · `3`=降噪;填子集如 `1,2` |
| | `denoise` | 文本 | `dereverb` | 第 3 段降噪法:`dereverb` / `deecho`(见 §5.2) |
| `/api/vocal` | `raw` | 文件 | (必填) | ① 原始清唱(跑调也行)→ 决定**唱什么内容** |
| | `ref` | 文件 | (必填) | ② 去和声参考(取目标旋律)→ 决定**唱成什么调** |
| | `self_ref` | 文件 | (必填) | ③ 你自己的清唱(音色锚 ~10–30s)→ 决定**听起来是谁** |
| | `correct_steps` | int | `150` | 修音步数(扩散迭代,50–200) |
| | `voice_steps` | int | `50` | 换音色步数(扩散迭代,20–100) |
| | `voice_cfg` | float | `0.7` | 人声近似度(cfg 引导强度,0–1) |

> 文件落盘规则(`_save_upload`):每个上传放进 `output/_uploads/<8位uuid>/<原文件名>` 独立子目录 —— 既保证唯一,
> 又**保留干净的原始文件名**(于是下游产物如 `twinkle.musicxml` 命名干净)。文件名只取 `basename` 防穿越。
> 缺文件返回 `None`,runner 会软失败提示「请先上传…」。

### 2.2 Job 轮询协议(`GET /api/job/{id}`)

返回 `Job.to_public()` 的 JSON(`web/jobs.py`):

```jsonc
{
  "id": "ab12…",
  "kind": "convert",          // convert / transcribe / separate / vocal
  "status": "running",        // queued / running / done / error  ← 终态:done / error
  "progress": 0.4,            // 0..1,粗粒度尽力而为(GPU 子进程内部不透明)
  "stage": "解析简谱 → 生成 MusicXML",  // 人类可读阶段文案,前端拿来刷新按钮
  "result": { … } | null,     // status=done 时为 runner 返回的结构化 dict(见 §2.3)
  "error":  "RuntimeError: …" | null,   // status=error 时的人话消息(硬错误)
  "created_at": 1730000000.0,
  "updated_at": 1730000005.2
}
```

- 未知 `job_id` → **HTTP 404** `{ detail: "未知 job_id" }`。
- 前端轮询节奏:`MM.poll` 每 **800ms** 拉一次,直到 `status` 为 `done` 或 `error` 才停。
- ⚠️ **任务存于进程内存**,无持久化:**重启服务后旧 job_id 全部 404**。这是单用户本地工具的取舍。

### 2.3 各 result 结构(逐字段;`status=done` 时)

通用约定:`result.ok`(布尔)= 业务是否成功;`result.message` = 人话消息;
`result.downloads` = `[{ name, size_kb, kind }]`(链接由 `MM.fileUrl(jobId, name)` 拼)。
`downloads[].name` 是**相对 job 目录的路径**(可能含子目录,如分离产物);`kind ∈ {score, midi, audio, data}`,
按扩展名归类(对应设计稿文件徽标配色)。

**convert** —— `run_convert`:

| 字段 | 类型 | 说明 |
|---|---|---|
| `ok` | bool | 成功/软失败 |
| `message` | str | 人话消息 |
| `direction` | str | `j2s` / `s2j`(前端据此决定渲染谱面还是简谱文本) |
| `staff_svg` | str \| null | `j2s` 时为五线谱 SVG **内联字符串**(直接 `innerHTML`);渲染失败为 `null`(不影响主产物) |
| `jianpu_text` | str \| null | `s2j` 时为简谱文本;`j2s` 时为 `null` |
| `downloads` | list | 全量产物(`.musicxml` / `.jianpu` / `staff.svg` …) |

**transcribe** —— `run_transcribe`:

| 字段 | 类型 | 说明 |
|---|---|---|
| `ok` | bool | 转录是否以 0 码成功 |
| `report_md` | str | `报告.md` 全文(**即便失败也带**,非零码会在开头加一行说明);软失败无报告时该字段可能缺 |
| `staff_svg` | str \| null | `staff.svg` 内联 SVG |
| `confidence_pct` | int(0–100) \| null | 整体可信度(从 `confidence.json` 尽力解析;读不到为 null) |
| `spots` | `[{ at, note }]` \| null | 存疑段(最多 12 条;`at`=时间区间文案,`note`=原因)|
| `downloads` | list | 优先按白名单顺序:`notes.json, out.mid, out.musicxml, jianpu.pdf, jianpu.ly, confidence.json, staff.svg`;白名单全落空则退回全量 |
| `message` | str | **仅软失败**(如引擎环境未就绪)时有 —— 此时无 `report_md` |

**separate** —— `run_separate`:

| 字段 | 类型 | 说明 |
|---|---|---|
| `ok` | bool | 成功/软失败 |
| `message` | str | 人话消息(软失败常见:本机无 `.venv-sep`) |
| `log` | str | 子进程日志尾部(成功取末 1200 字,失败取末 2000 字;无则 `""`) |
| `tracks` | `[{ name, size_kb, kind }]` | 分离出的各轨 wav(同时也是 downloads;均为 `audio`) |

**vocal** —— `run_vocal`:

| 字段 | 类型 | 说明 |
|---|---|---|
| `ok` | bool | 是否产出成品 |
| `message` | str | 人话消息 |
| `detail` | str | 各阶段产物明细(成功)或 `traceback` 尾部(异常) |
| `final` | str \| null | 成品 wav 的**相对 job 目录路径**;无成品为 null |
| `mid` | str \| null | 中间产物(只修了音准的 24k wav)相对路径 |
| `downloads` | list | 全量产物 |

### 2.4 下载 / 预览 URL(`GET /api/file/{id}/{name}`)

- URL 形如 `/api/file/<job_id>/<相对路径>`;`name` 可含子目录(`name:path`)。
- 前端用 `MM.fileUrl(jobId, name)` 生成:对 `job_id` 与 `name` 的**每一段**分别 `encodeURIComponent`(保留 `/`),
  兼容中文 / 特殊字符文件名。
- **防穿越**:`JobManager.file_path` 把 `(base/name)` resolve 后强制 `relative_to(job_dir)`,`..` 越界返回 404。
- 响应是 `FileResponse`,带 `filename=basename`;浏览器既可直接 `<a download>`,也可塞给 `<audio src>` 预览播放。

## 3. 任务模型(`web/jobs.py`)

### 3.1 为什么是「提交 → 轮询 → 下载」

拆声(分钟级)和重塑(整首高精度可 20+ 分钟)远超一次 HTTP 请求的合理时长,所以不能同步返回结果。
互译/记谱虽快,也**走同一套异步路径**(首次轮询往往就是 `done`),这样前端逻辑对四个部门完全一致。

### 3.2 三步流转

1. **提交**:`POST /api/<fn>` → `JM.submit(kind, runner, *args)` 建 `Job`(状态 `queued`)、排进线程池、立即返回 `job_id`。
   每个 job 分到独立目录 `output/<kind>_<毫秒时间戳>_<6位uuid>/`,产物都写这里。
2. **执行**:worker 线程把 job 置 `running`,调 `runner(job, *args)`。runner 可随时 `job.set_stage(文案, progress)`
   更新进度(线程安全)。
3. **轮询 + 下载**:前端每 800ms 拉 `/api/job/<id>`;终态后据 `result.downloads` / `final` / `tracks` 拼
   `/api/file/<id>/<name>` 下载或喂播放器。

### 3.3 软失败 vs 硬错误(前端必须区分的两条路)

| | 软失败 | 硬错误 |
|---|---|---|
| 触发 | runner **正常返回** dict 且 `result.ok === false` | runner **抛异常** |
| `job.status` | `done` | `error` |
| 典型场景 | 没传文件、GPU venv/权重未就绪、子进程非零码 | 代码 bug、意料外崩溃 |
| 信息位置 | `job.result.message`(+ 可能的 `log`/`detail`/`report_md`) | `job.error`(`"类型: 消息"`;完整 traceback 打到服务端控制台) |
| 前端应对 | 友好展示 message(**软失败也可能有部分产物 / 报告,要尽量显示**) | toast 报错即可 |

> 这条边界是整个前端的「契约核心」:**`ok=false` 不是崩,是『环境/输入没就绪』的可预期结果**,
> 用户要看到怎么补救;只有 `status=error` 才是真 bug。runner 把所有可预期失败都收成 `ok=false`,
> 真异常才上抛由 `JobManager` 兜底(写 `status=error` + 控制台打 traceback,worker 线程绝不静默死掉)。

## 4. 前端结构

```
musicmaster/web/static/
├─ index.html              ★ 设计稿:四面板 DOM、黑胶唱机/谱面/可信度环等视觉、底部脚本(Tab/唱机/旋钮交互)
├─ fonts/                  自托管 OFL 字体(Fraunces/Inter/JetBrains Mono;无外链、离线、不泄露访客 IP)
└─ js/
   ├─ api.js               ★ window.MM 共享客户端(提交/轮询/下载/播放器/toast 全部 helper)
   ├─ convert.js           互译部门:接 #p-convert
   ├─ transcribe.js        记谱部门:接 #p-transcribe
   ├─ separate.js          拆声部门:接 #p-separate
   └─ vocal.js             重塑部门:接 #p-vocal
```

加载顺序(`index.html` body 末尾):**先**一段内联脚本(纯设计稿交互:主 Tab 切换 + 主色、产出子标签、
唱机播放 mockup、旋钮数值、拆声清理药丸切换、拖拽视觉),**再**依次 `api.js → convert/transcribe/separate/vocal.js`。
部门模块在 DOM 与 `window.MM` 均就绪后才接线,各自独立文件(避免并行接入互相冲突)。

### 4.1 设计稿 `index.html`(部门 JS 依赖的 DOM 钩子)

设计稿本身**不接任何后端**,只提供结构与视觉;部门 JS 靠下列稳定选择器把真实数据填进去:

| 钩子 | 含义 |
|---|---|
| `#p-convert` `#p-transcribe` `#p-separate` `#p-vocal` | 四个面板根(`MM.panel(fn)` = `getElementById('p-'+fn)`) |
| `.console[data-accent]` + `.tab[data-c]` | 主 Tab 切换 + 每部门主色(内联脚本负责;部门 JS 不碰) |
| `[data-outtabs] .out-tab` + `.out-body .out-pane` | 「产出」子标签与对应面板(`MM.switchOut(panel, idx)` 模拟点击切换) |
| `.cast` | 该面板的提交按钮(`MM.setBusy` 接管忙碌态) |
| `.drop`(内含 `<input type=file>`) | 拖放/选择上传区(互译、重塑用;`MM.makeDrop`) |
| `[data-player]`(黑胶唱机) | 既当「上传+预览」控件(`MM.makeUploadPlayer`,选文件后即可真声试听),又当结果播放器(`MM.wirePlayer`) |
| `.dl-grid` | 「带走」下载网格容器(`MM.renderDownloads`) |
| 互译:`input[name=cv]:checked`、`.input` | 方向单选、主音输入 |
| 记谱:`input[name=tr]:checked`、`.input`、`.ring`/`.spot`/`pre[data-report]` | 引擎单选、调、可信度环/存疑段/报告 |
| 拆声:`.steps .step input[checkbox]`、`.cm-pills .mini.on[data-val]`、`.donelist` | 处理段勾选、清理模型药丸、成品清单 |
| 重塑:`.vc-ins .drop`(三个,按序 raw/ref/self_ref)、`[data-slider=cs|vs|cfg]`、`.player.showpiece` | 三上传、三旋钮、成品大唱机 |

> **设计原则(改前端务必遵守)**:① 部门 JS **只接线、不造 UI**,所有渲染走 `MM`;② 真实数据进来后**清掉
> mockup 假数据**(如互译把 `.facts` 的假「6 小节 42 音符」清空隐藏);③ 只改对应部门的 `*.js`,**不动**
> `index.html` / `api.js` / 其它部门模块(除非真要加新面板,见 §5)。

### 4.2 `api.js` —— `window.MM` helper 清单

所有部门共享。按用途分组:

**网络 / 任务**
| helper | 签名 | 用途 |
|---|---|---|

### 4.3 四个部门 JS 模块各做什么

四者结构高度一致:`init()` 里取面板与控件 → 给 `.cast` 绑 click → `MM.run(...)` → 在 `.then(job)` 里分流
(硬错误 / 软失败 / 成功)→ 渲染 → `.finally(restore)`。差异只在「读哪些输入」「渲染哪些 pane」。

- **convert.js**(`#p-convert`,产出 pane:0 誊写 / 1 谱面 / 2 带走)
  读 `.drop` 文件 + `input[name=cv]` 方向 + `.input` 主音 → 提交 `file/direction/key`。
  渲染:誊写盖章 + 清 mockup 假统计;`j2s` 把 `staff_svg` 注入 `.sheet`,`s2j` 把 `jianpu_text` 放进 `.jianpu` 的 `<pre>`;
  带走渲染下载。有五线谱预览就跳「谱面」,否则停「誊写」。

- **transcribe.js**(`#p-transcribe`,pane:0 解读 / 1 谱面 / 2 带走)
  读唱机文件 + `input[name=tr]` 引擎 + 调 → 提交 `audio/engine/key`。
  渲染:可信度环(`confidence_pct` 驱动 `.ring` 的 `stroke-dashoffset`)+ 概述 + 存疑段(`spots` 重建 `.spot`)+
  **报告全文始终展示**(复用同一 `<pre data-report>`)。**关键**:即便 `ok===false`(非零码失败)只要有 `report_md`
  仍完整展示报告/谱面/产物,绝不静默;只有「软失败且无报告」(引擎环境未就绪)才仅 toast + 概述显示 message。

- **separate.js**(`#p-separate`,pane:0 几束声音 / 1 带走)
  读唱机文件 + `collectStages()`(按勾选的第 i 个映射成序号 `i+1`,join `,`)+ `readDenoise()`(读高亮药丸 `data-val`)
  → 提交 `audio/stages/denoise`。
  渲染:以 pane0 内第一个唱机为**模板**,按 `tracks` 克隆出每束音频的唱机并 `wirePlayer` 接真实播放;更新成品清单;
  带走渲染下载。软失败把 message 写进面板 + toast,并清空带走网格避免残留 mockup。

- **vocal.js**(`#p-vocal`)
  读三个 `.vc-ins .drop`(按序 raw/ref/self_ref,**语义不能搞反**,逐个校验缺失给精准提示)+ 三旋钮 `.value`
  → 提交六字段。面板原本无下载区,模块在按钮所在卡片末尾 `ensureGrid()` 动态建一个 `.dl-grid`。
  渲染:成品 → `.player.showpiece`,中间产物 → 另一个唱机,下载网格。**软失败也尽量展示**中间产物/可下载文件。

## 5. 如何扩展(新增一个功能页签)

新增「第五个功能」要动**四处**(后端两处 + 前端两处),彼此解耦、照抄现有部门即可:

**① runner(`web/runners.py`)** —— 写 `run_<fn>(job, *args) -> dict`:
```python
def run_myfeat(job, audio_path: str, opt: str = "x") -> dict:
    if not audio_path:
        return {"ok": False, "message": "请先上传文件。"}   # 软失败
    d = Path(job.job_dir)
    job.set_stage("处理中…", 0.3)                            # 更新进度
    try:
        # 复用核心模块(本进程 import)或子进程(GPU venv:仿 run_separate 的 _sep_python + subprocess)
        ...
        return {"ok": True, "message": "完成", "downloads": _downloads(d)}
    except Exception as e:                                   # 可预期失败收成软失败
        return {"ok": False, "message": f"失败:{type(e).__name__}: {e}"}
```
返回的 dict 即 `job.result`;产物写进 `job.job_dir`,用 `_downloads(d)` 列出。真异常可不 catch,交 `JobManager`。

**② 端点(`web/server.py`)** —— 加一个 `POST /api/myfeat`(**写在 `app.mount` 之前**):
```python
@app.post("/api/myfeat")
def api_myfeat(audio: UploadFile = File(...), opt: str = Form("x")) -> dict:
    path = _save_upload(audio)
    job = JM.submit("myfeat", runners.run_myfeat, path, opt)
    return {"job_id": job.id}
```

**③ 部门 JS(`web/static/js/myfeat.js`)** —— 照抄一个最接近的部门模块,改成读自己的输入、`MM.run('myfeat', fd, …)`、
渲染自己的 result 字段。**只用 `MM` helper,不造新 UI 原语。**

**④ 设计稿面板(`index.html`)** —— 加 `<button class="tab" data-c="myfeat">` 进 `.tabs`,加 `<div class="panel" id="p-myfeat">`
进 `.panels`,在 `map`(底部内联脚本里 `{convert:'p-convert', …}`)加一行 `myfeat:'p-myfeat'`,并在 body 末尾
`<script src="js/myfeat.js">`。若要部门主色,加 `.console[data-accent="myfeat"]{…}` 与 `.tab[data-c="myfeat"]{--ac:…}`。

> 改完后端要**重启服务**(模块已加载进进程,不重启不生效);改 JS/HTML 只需刷新浏览器。

## 6. venv 路由(谁在哪个环境跑)

四类重依赖互相冲突,**分多个 venv**;runner 据功能路由(与历史 `app.py` 完全一致):

| 功能 | 跑在哪 | 怎么调 |
|---|---|---|
| **convert / transcribe** | **本进程**(`.venv`,CPU) | runner 内**直接 `import`** 调核心模块 |
| **separate** | **子进程** → 分离 venv(`.venv-sep`,GPU) | `run_separate` 用 `_sep_python()` 找到其 `python.exe`,`subprocess.run(... -m musicmaster.separate.pipeline ...)`,超时 1800s |
| **vocal** | 本进程调 `vocal.pipeline.two_stage` | `two_stage` **内部再起 GPU 子进程**(`.venv-neural` 修音 + `.venv-svc` 换音色) |

**`.venv-sep` 的 python 怎么找**(`_sep_python`,优先级):
环境变量 `MUSICMASTER_SEP_PYTHON` > `paths.local.json` 的 `sep_python` 键 > vendor 默认
(`vendor/.venv-sep/Scripts/python.exe` 或 `./.venv-sep/Scripts/python.exe`)。找不到 → 软失败提示装环境。

**子进程环境**(`_subprocess_env`):带 `PYTHONUTF8=1` + `PYTHONIOENCODING=utf-8`(防 GBK 乱码)+ 把仓库根塞进 `PYTHONPATH`。

**GPU 路径配置全表**(env > `paths.local.json` > 默认;详见 [`架构流程图.md`]):

| 环境变量 | `paths.local.json` 键 | 指向 |
|---|---|---|
| `MUSICMASTER_SEP_PYTHON` | `sep_python` | 分离 venv 的 python |
| `MUSICMASTER_VOCAL_PYTHON` | `vocal_python` | `.venv-neural`(修音)的 python |
| `MUSICMASTER_SVC_PYTHON` | `svc_python` | `.venv-svc`(换音色)的 python |
| `MUSICMASTER_DIFFPITCHER_DIR` | `diffpitcher_dir` | DiffPitcher 源码目录 |
| `MUSICMASTER_SEEDVC_DIR` | `seedvc_dir` | seed-vc 源码目录 |
| `MUSICMASTER_SEP_MODELS` | — | 分离权重缓存目录 |
| `LILYPOND_EXE` | — | LilyPond 可执行(简谱出 PDF 需要;**勿入全局 PATH**,DLL 冲突) |

- `paths.local.json` **不入库**(gitignore);Windows 路径用**正斜杠 `/`** 或双反斜杠(单反斜杠会让 JSON 解析失败)。
- 没装 GPU venv 时,拆声/重塑页给友好**软失败**提示,**不影响互译/记谱**。简谱出 PDF 需 LilyPond,没装则只产 `.ly`。

## 7. 模型 / 引擎对比表(讲给用户的版本)

> 专业名词保留,但要让用户知道**区别**与**何时选哪个**。可做成各面板的可展开「帮助」弹层
> (设计稿里已有 `details.help` 结构与简版文案)。默认值(crepe / dereverb / 150 / 50 / 0.7)对绝大多数人最优。

### 7.1 记谱引擎(`engine`)

| 引擎 | 中文定位 | 适合素材 | 出简谱? | 依赖 | 坑 |
|---|---|---|---|---|---|
| **crepe**(默认) | 单旋律深度学习音高跟踪(CREPE/torchcrepe) | 人声清唱、哼唱、单线条乐器 | ✅ 出简谱 + 逐音可信度 | TF 2.15(本进程 CPU) | 复音(和弦)会丢音 |
| **basic-pitch** | Spotify 通用复音音符检测 | 多音/和弦/乐器、兜底 | ❌ 仅五线谱 | TF 2.15(干净环境) | 噪声下人声主旋律会崩到 ~12%,**人声别用它** |
| **bytedance** | ByteDance 钢琴专用高分辨率(SOTA) | 干净 44.1k **真钢琴**录音 | ❌ 仅五线谱 | torch + GPU + 首次下权重 | 低质/非钢琴是 OOD,**会幻觉假和弦** |

**一句话给用户**:唱的/哼的/单旋律 → `crepe`;多个音同时响(和弦/伴奏)→ `basic-pitch`;纯钢琴干净录音 → `bytedance`。
**选错不会崩**(已加兜底 + 入口质量门拦烂输入 + 报告里给提示),但结果会差。

> 三层质量护栏(记谱独有,体现在 `report_md` 与 `confidence_pct`/`spots`):
> **L1 入口质量门**(转录前体检录音真实带宽,烂输入门口拦+给修法)→ 选引擎转录 →
> **L3 逐音可信度**(CREPE 与 pyin 两种听法交叉验证,成串调外/跟踪自信度低 → 标「存疑段」)。
> 调外 ≠ 错(真曲也有变化音),所以只标「存疑」,从不武断判错。

### 7.2 拆声三段级联 与 降噪法(`stages` / `denoise`)

三段级联(`stages` 选子集):
① **BS-RoFormer** 分离人声/伴奏 → ② **Karaoke RoFormer** 去和声(去 backing vocals)→ ③ 降噪提纯。
普通分离只到 ①;**「去 backing vocals、只留主唱」靠 ②**(本项目特色);③ 才是下面的 dereverb/deecho。

| `denoise` | 作用 | 何时用 |
|---|---|---|
| **dereverb**(默认) | 去混响(房间残响),取「No Reverb」干声轨;默认模型 UVR-DeEcho-DeReverb 已能同时压残响+回声 | 大多数情况 |
| **deecho** | 去回声(重复回声),取「No Echo」轨 | 原录音回声特别重时 |

### 7.3 重塑:三输入 + 三旋钮

⭐ **三个输入是产品核心设计,别退回两输入**(原因见 `合并开发日志.md`):

| 输入 | 决定 | 要求 |
|---|---|---|
| ① `raw` 原始清唱 | 唱**什么内容** | 跑调也行 |
| ② `ref` 去和声参考 | 唱成**什么调**(目标 f0) | **必须去和声**(可先在「拆声」对原曲跑出 clean/lead 当 ②);且与 ① **同段、逐句时间对齐**(总时长、每句起止、节奏型一致) |
| ③ `self_ref` 你的清唱 | 听起来**是谁**(音色) | ~10–30s 干净清唱(超 30s 自动截取) |

| 旋钮(字段) | 是什么 | 越大 | 默认 | 给用户的话 |
|---|---|---|---|---|
| 音准精修(`correct_steps`) | DiffPitcher 扩散迭代 | 音准修得越精细,越慢 | 150 | 想快可降到 50 |
| 音色细腻度(`voice_steps`) | Seed-VC 扩散迭代 | 音质越细腻,越慢 | 50 | 25 也够用 |
| 人声近似度(`voice_cfg`) | Seed-VC 分类器自由引导强度 | 越强调「贴目标音色」条件 | 0.7 | **效果较细微**;设 0 约快 1.5× |

> ⚠️ **诚实告知**:本管线用 Seed-VC **V1**,其 cfg 据官方是「subtle(细微)」级旋钮;真正强力的相似度旋钮
> (`similarity-cfg-rate`)只在 **V2** 才有,本项目**未上 V2**。**想更像本人**:优先换更干净/更长的 ③ 音色锚 +
> 适当加大「音色细腻度」,效果比拉 cfg 明显。整首会**自动分块出完整长度**(不截断),但**时长×步数越大越慢**
> (整首×150 步在 8GB 上约 20+ 分钟,先拿副歌试更快)。

## 8. 常见坑

1. **`core/__init__` 同名陷阱(本仓头号反复 bug)**:`musicmaster/core/__init__.py` 把
   `render`/`transcribe`/`separate`/`correct` 这些**函数**绑成了包属性。所以
   `from musicmaster.core import render` 拿到的是**函数**,`render.render(...)` 会 `AttributeError`。
   **正解**:`from musicmaster.core.render import render`(直接从子模块)。`vocal/__init__` 同理。
   `run_convert` 渲染五线谱预览时就是这么做的。

2. **GBK 控制台 `reconfigure`**:Windows 默认代码页是 GBK,打印含中文/emoji 会 `UnicodeEncodeError`。
   `server.py` / `runners.py` / `jobs.py` 都在顶部 `sys.stdout/stderr.reconfigure("utf-8")`;子进程靠
   `_subprocess_env` 带 `PYTHONUTF8=1`/`PYTHONIOENCODING=utf-8`;启动器靠 `chcp 65001` / `PYTHONUTF8=1`。
   新增打印代码沿用,别裸 `print` 中文到未配置的控制台。

3. **长任务进度不透明**:GPU 子进程(拆声/重塑)内部进度拿不到,`progress`/`stage` 只能**粗粒度**更新
   (runner 在关键节点 `set_stage`)。前端必须有「进行中」态(`MM.setBusy` + `onProgress` 刷阶段文案),
   **别让用户以为卡死**;重塑尤其要提示「整首高精度可能 20+ 分钟」。轮询 800ms 一拍即可,别更密。

4. **上传命名**:上传文件落进 `output/_uploads/<uuid>/<原名>`(独立子目录而非加前缀),为的是下游产物
   命名干净。文件名只取 `basename`(防穿越)。前端展示下载项时也只显 basename(`MM.renderDownloads` 已处理),
   但 `MM.fileUrl` 必须传**完整相对路径**(可能含子目录),否则 404。

5. **静态站挂载顺序**:`app.mount("/", StaticFiles…)` 在根,会吞掉它之后注册的所有路由 —— **新 `/api` 路由
   必须加在 `mount` 之前**(§5)。

6. **改后端要重启服务**:Python 模块已加载进运行进程,不重启不生效(改 JS/HTML 只需刷新浏览器)。

7. **任务无持久化**:job 存进程内存,**重启服务后旧 job_id 全 404**;`output/` 里的产物文件仍在,但要靠新 job 才能经
   `/api/file` 取(因为路径解析以 job 目录为根)。这是单用户本地工具的取舍,不是 bug。
